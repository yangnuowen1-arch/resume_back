# 邮箱导入简历 → AI 解析 → 回填候选人基础信息

## 目标
mailbox 导入 resume 后（已自动上传 R2），异步调用新建的 Dify「简历解析」workflow，从简历提取结构化基础信息，回填到 candidate 表对应字段。同时把简历纯文本写入 `resume.raw_text`、结构化 JSON 写入 `resume.parsed_data`、`parse_status` 置为 `parsed`。

**不需要数据库迁移**（candidate/resume 字段都已存在），**不需要新建任务表**（复用 resume.parse_status 作为任务状态）。

---

## 一、Dify 平台侧（你来做，代码侧配合）
在 Dify 新建一个 Workflow 类型 App「简历解析」：
- **输入**：`resume_file`（文件/document）、`output_language`（字符串，默认 "Chinese"）—— 与 screening 同名输入，复用配置
- **输出变量**：`parse_result`（字符串，JSON），结构如下：
```json
{
  "name": "张三",
  "phone": "138****",
  "email": null,
  "gender": "男",
  "current_company": "字节跳动",
  "current_position": "Go 后端工程师",
  "years_of_experience": 5,
  "highest_education": "本科",
  "school": "某某大学",
  "major": "计算机科学",
  "location": "北京",
  "raw_text": "简历全文纯文本..."
}
```
- 拿到这个 App 的 API key，填入 `.env` 的 `DIFY_PARSE_API_KEY`
- 字段映射到 candidate 表的 10 个字段（candidate 表无 skills 字段，故不提取技能；若想要可后续存在 parsed_data 里）

## 二、代码改动（按文件）

### 1. `internal/config/config.go` — 新增解析 workflow 配置
Config 结构体加 3 个字段：
- `DifyParseAPIKey`（env `DIFY_PARSE_API_KEY`，默认空）
- `DifyParseResultOutputName`（env `DIFY_PARSE_RESULT_OUTPUT_NAME`，默认 `parse_result`）
- `DifyParseWorkerCount`（env `DIFY_PARSE_WORKER_COUNT`，默认 2）

LoadConfig 里读取。新增 `DifyParseEnabled()` 方法 = `DifyParseAPIKey != ""`（镜像 `DifyEnabled()`）。
复用已有的 `DifyBaseURL`/`DifyUser`/`DifyResumeFileInputName`/`DifyOutputLanguageInputName`。

### 2. `internal/dify/client.go` — 新增 `ExtractResumeInfo` 方法
现状：`Client` 单 apiKey，`runWorkflow` 硬编码 screening 的 inputs（resume_file + job_context + output_language）。

改动：
- 泛化 `uploadFile` 与 `runWorkflow`：加一个 `apiKey string` 和 `inputs map[string]interface{}` 参数（保留原有重试/HTTP 逻辑不变）。`RunResumeScreening` 改为传入 `c.apiKey` 和它自己的 inputs map（行为不变）。
- 新增类型 `ExtractResumeInfoRequest{File, Filename, ContentType, OutputLanguage, User}` 和 `ExtractResumeInfoResponse{WorkflowRunID, TaskID, ResultText, RawResponse, RetryCount}`。
- 新增方法 `(c *Client) ExtractResumeInfo(ctx, req)`：用 `c.parseApiKey` + `c.parseResultOutputName`，inputs 只含 `resume_file` + `output_language`（无 job_context），调 `uploadFile` + 泛化的 `runWorkflow`。
- `Config` 加 `ParseAPIKey`、`ParseResultOutputName` 字段；`NewClient` 填充 `c.parseApiKey`/`c.parseResultOutputName`。

### 3. `internal/service/resume_parse_service.go` — 新建（核心，镜像 screening_task_service）
定义：
```go
type ResumeParseEnqueuer interface {
    EnqueueParse(ctx context.Context, resumeID int64) error
}
type DifyResumeParseClient interface {
    ExtractResumeInfo(ctx, dify.ExtractResumeInfoRequest) (*dify.ExtractResumeInfoResponse, error)
}
type ResumeParsedInfo struct {
    Name, Phone, Email, Gender, CurrentCompany, CurrentPosition *string
    YearsOfExperience *float64
    HighestEducation, School, Major, Location, RawText *string
}
```
`ResumeParseService` 结构体持有：`resumeRepo`、`candidateRepo`、`uploader`、`difyParseClient`、`difyUser`、内存 `queue chan int64`。
- `NewResumeParseService(deps)`：若依赖齐全则建队列 + 启 worker（镜像 `screeningDependenciesConfigured` / `startScreeningWorkers`）。
- `EnqueueParse(ctx, resumeID)`：校验 ID，push resumeID 到 channel；队列满返回错误（镜像 `enqueueScreeningJob`）。
- `runParseWorker`：消费 channel → `processQueuedParse`（镜像 `runScreeningWorker`）。
- `processQueuedParse(ctx, resumeID)`：
  1. `resumeRepo.MarkParsing(resumeID)`
  2. `resumeRepo.FindByID` → `resumeObjectKey(resume)`（复用现有 `screening_task_service.go` 里同名的 key 反解函数，它已是包级函数）→ `uploader.Open` 读文件
  3. `difyParseClient.ExtractResumeInfo(...)` 传文件流
  4. `parseResumeAIOutput(resultText)` 解析 JSON 成 `ResumeParsedInfo`（复用 `stripJSONCodeFence`）
  5. 成功：`resumeRepo.MarkParsed(resumeID, rawText, parsedDataJSON, language, now)` 写回 resume；再 `candidateRepo.UpdateBasicInfoFromParse(resume.CandidateID, info)` 回填 candidate
  6. 失败：`resumeRepo.MarkParseFailed(resumeID, message)`
- 常量：`defaultResumeParseQueueSize=100`、`defaultResumeParseWorkerCount=2`、`maxResumeParseWorkerCount=10`。

### 4. `internal/repository/candidate_repository.go` — 新增回填方法
接口加 `UpdateBasicInfoFromParse(ctx, candidateID int64, info service.ResumeParsedInfo) error`。
实现用一条带 `COALESCE` 的 UPDATE（单事务），策略：
- `name = COALESCE(?, name)` —— **姓名覆盖**（mailbox 派生的名字是文件名/邮箱前缀，质量低，AI 提取的真名更好）
- `phone = COALESCE(phone, ?)`、`gender = COALESCE(gender, ?)`、`current_company = COALESCE(current_company, ?)`、`current_position = COALESCE(current_position, ?)`、`years_of_experience = COALESCE(years_of_experience, ?)`、`highest_education = COALESCE(highest_education, ?)`、`school = COALESCE(school, ?)`、`major = COALESCE(major, ?)`、`location = COALESCE(location, ?)` —— **仅填空**（不覆盖人工已填的数据，保护已有候选人）
- `email` **不动**（是去重唯一键 + 已从发件人正确填入）
- 用 `sql.Named` 或显式 `Exec` 构造；nil 指针传 `nil` 让 COALESCE 跳过。

> 注：`ResumeParsedInfo` 定义在 service 包，repository 引用 service 包会循环依赖。解决：把 `ResumeParsedInfo` 放到 `internal/dto`（和其它 DTO 一致），service 和 repository 都引用 dto。或放 repository 包。实现时选 dto。

## 三、端到端数据流
```
mailbox 扫描 → importAttachments
  ├─ uploader.UploadBytes → R2（已完成）
  ├─ buildCandidate/CreateWithResume → candidate(pending_review) + resume(parse_status=pending)
  └─ resumeParseEnqueuer.EnqueueParse(resumeID) → 内存队列  [新增]
                                                    │
       ResumeParseService worker 消费  [新增]
         ├─ MarkParsing
         ├─ uploader.Open(resumes/<uuid>.pdf) 读 R2 文件
         ├─ Dify.ExtractResumeInfo(文件流) → parse_result JSON
         ├─ parseResumeAIOutput → ResumeParsedInfo
         ├─ resumeRepo.MarkParsed(raw_text, parsed_data, parse_status=parsed)  [回写 resume]
         └─ candidateRepo.UpdateBasicInfoFromParse  [回填 candidate：name 覆盖，其余填空]
```

## 四、关键设计决策
1. **复用 resume.parse_status 当任务状态**，不新建表 —— 与 screening 用独立表不同，但 resume 本身就是天然载体，更简单。
2. **channel 驱动，不轮询** —— 只解析导入时入队的 resume，不会误碰用户手动 parse 的简历，无并发冲突。
3. **回填策略**：name 覆盖、email 不动、其余字段仅填空 —— 保护人工数据，同时修正低质量的派生姓名。
4. **降级安全**：Dify 解析未配置/失败都不阻断导入；解析失败 resume 标 `failed`，可走现有 `POST /resumes/:id/parse` 手动重试（或后续加重解析接口）。
5. **bonus**：解析顺带填 `resume.raw_text`，让后续 screening 的 evidence 定位（`buildRequirementsJSON` 依赖 resumeText）也能工作。
